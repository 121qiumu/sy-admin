## 真实接口对接清单

### 1. 获取验证码

- 方法：`GET`
- 路径：`/admin/base/open/captcha`
- 是否带 token：否
- 前端用途：登录页初始化验证码、登录失败后刷新验证码

### 2. 登录

- 方法：`POST`
- 路径：`/admin/base/open/login`
- 是否带 token：否
- 请求字段：
  - `username`
  - `password`
  - `captchaId`
  - `verifyCode`

与 Step 1 约定保持一致：

- `verifyCode` 按字符串提交
- 不带 `Bearer`

### 3. 获取当前用户信息

- 方法：`GET`
- 路径：`/admin/base/comm/person`
- 是否带 token：是
- 请求头：`Authorization`

前端用途：

- 登录成功后初始化用户信息
- 刷新页面后恢复当前登录用户状态

### 4. 退出登录

- 方法：`POST`
- 路径：`/admin/base/comm/logout`
- 是否带 token：是

前端策略：

- 先尝试请求后端退出
- 无论接口是否报错，最终都清理本地登录态

## 路由与白名单处理

### 白名单路由

当前白名单：

- `/login`
- `/404`

### 守卫策略

#### 未登录访问受保护页面

- 自动跳转到 `/login`
- 带上 `redirect` 参数

#### 已登录访问登录页

- 若用户信息可正常恢复，则自动跳回目标页或 `/dashboard`

#### 已登录访问受保护页面

- 允许进入
- 应用启动后自动补拉当前用户信息

## 登录成功后的跳转逻辑

当前规则：

1. 登录成功后先存储 token 与 refreshToken
2. 再调用 `/admin/base/comm/person`
3. 获取当前用户信息成功后跳转
4. 若路由 query 中存在合法 `redirect`，优先回跳
5. 否则默认跳转 `/dashboard`

## 退出登录逻辑

当前流程：

1. 顶部栏点击退出登录
2. 二次确认
3. 调用 `/admin/base/comm/logout`
4. 清空本地 token、refreshToken、userInfo
5. 跳转回 `/login`

## 登录失效处理

本步继续复用 Step 3 请求层能力：

- 请求拦截器自动注入 `Authorization`
- 响应拦截器统一处理 `401`
- `401` 时自动清理登录态并跳登录页

这与 Step 1 的接口约定保持一致：

- 前端不能只看业务 `code`
- 需要同时处理 HTTP 状态码

## 真实联调结果

### 1. 验证码接口

实测接口：

- `GET http://127.0.0.1:8001/admin/base/open/captcha`

结果：

- 返回 `code = 1000`
- 返回体包含 `captchaId`
- 返回体包含 `data:image/svg+xml;base64,...`

结论：

- 登录页可以直接使用真实验证码接口

### 2. 当前用户信息接口的未登录拦截

实测接口：

- `GET http://127.0.0.1:8001/admin/base/comm/person`

无 token 结果：

- HTTP 状态：`401`
- 响应体：`{"code":1001,"message":"登录失效或无权限访问~"}`

结论：

- 受保护接口的登录失效处理与文档约定一致
- 前端请求层统一处理 `401` 是正确方向
