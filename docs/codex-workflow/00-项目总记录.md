## Step 1 结论

### 1. 路由规则

- 管理后台接口统一位于 `/admin/...`
- 控制器路径与接口前缀存在稳定映射关系
- `api: ['add', 'delete', 'update', 'info', 'list', 'page']` 会自动生成标准 CRUD 接口

确认状态：

- 路由前缀规则：`已确认`
- 每个 CRUD 动作的请求方式：`已确认`

### 2. 鉴权规则

- 管理后台 token 通过请求头 `Authorization` 传递
- 传递的是原始 token 字符串
- 不是 `Bearer <token>`

确认状态：

- 请求头名称：`已确认`
- token 值格式：`已确认`

### 3. 响应规范

绝大多数接口遵循：

```json
{
  "code": 1000,
  "message": "success",
  "data": {}
}
```

常见 code：

- `1000`：成功
- `1001`：通用业务失败
- `1002`：参数校验失败
- `1003`：核心失败

分页数据结构：

```json
{
  "list": [],
  "pagination": {
    "page": 1,
    "size": 15,
    "total": 0
  }
}
```

确认状态：

- `code/message/data` 结构：`已确认`
- 默认分页字段：`已确认`
- 默认分页大小 `15`：`已确认`

- 本地开发配置里 `cool.eps = true`
- 生产配置里 `cool.eps = false`

确认状态：

- 配置差异：`已确认`
- 前端不应强依赖：`已确认`

## Step 1 关键提醒

- `verifyCode` 在 DTO 声明里是 `number`，但服务层按字符串验证码处理
- `/admin/base/open/html` 与 `/admin/base/sys/param/html` 返回原始 HTML，不是 JSON 包装
- `/admin/base/comm/upload` 成功时 `data` 是字符串 URL，不是对象
- `dict/info/data` 需要登录，但权限校验放宽
- `/admin/task/info/once`、`/start`、`/stop` 控制器里没有显式 `return`，返回体形态要在联调时再次验证
- 项目中未发现专门的文件下载接口，现阶段按“上传后返回 URL 直接访问”处理
