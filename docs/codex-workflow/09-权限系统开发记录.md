## 真实接口对接清单

### 1. 获取菜单与权限

- 方法：`GET`
- 路径：`/admin/base/comm/permmenu`
- 是否带 token：是
- 请求头：`Authorization`

前端用途：

- 初始化当前用户可见菜单
- 初始化当前用户可访问路由
- 初始化当前用户按钮权限码

### 2. 当前接口返回结构要点

根据 Step 1 文档与后端实现，本步按以下真实约定落地：

- 返回体包含 `menus`
- 返回体包含 `perms`
- `menus` 是平铺数组，不是树结构
- `perms` 是当前用户最终可用权限码数组

### 3. 菜单字段使用方式

本步前端实际消费的字段：

- `id`
- `parentId`
- `name`
- `router`
- `perms`
- `type`
- `icon`
- `orderNum`
- `viewPath`
- `keepAlive`
- `isShow`

前端映射规则：

- `parentId`：用于组装菜单树
- `router`：作为页面路由路径
- `perms`：作为按钮权限码来源
- `type = 0`：目录
- `type = 1`：页面菜单
- `type = 2`：按钮权限点
- `icon`：映射为侧边栏图标
- `orderNum`：用于菜单排序
- `viewPath`：当前先保存在路由 `meta` 中，后续真实业务页面替换时使用
- `keepAlive`：映射到路由缓存标记
- `isShow`：控制菜单是否在侧边栏展示

## 权限架构实现说明

### 1. 权限数据单一来源

当前权限系统统一以 `/admin/base/comm/permmenu` 的返回结果为准：

- `menus` 决定菜单与可访问页面
- `perms` 决定按钮权限点

前端不再手写静态业务菜单，也不在多个地方重复维护权限码。

### 2. 权限状态管理

新增 `src/stores/modules/permission.js`，集中管理：

- `perms`
- `rawMenus`
- `menuTree`
- `sidebarMenus`
- `accessiblePaths`
- `dynamicRouteNames`
- `routesLoaded`

当前 store 能力：

- `setPermissionData`
- `resetPermissionState`
- `ensurePermissionRoutes`
- `hasPermission`
- `canAccessPath`
- `isKnownProtectedPath`
- `resolveOpenKeys`

### 3. 菜单树与动态路由转换

新增 `src/permission/index.js`，负责：

- 平铺菜单转树
- 目录、页面、按钮三类节点识别
- 侧边栏菜单过滤
- 动态路由生成
- 可访问路径提取
- 菜单展开 key 计算

当前策略：

- 目录节点可生成重定向路由
- 页面节点统一映射到通用权限占位页
- 按钮节点不生成页面路由，只用于按钮权限控制

说明：

- 本步不提前接入真实业务页面组件
- 页面节点暂时统一落到 `src/views/permission/workspace.vue`
- 后续做真实业务模块时，再按 `viewPath` 或模块规划替换为真实页面

### 4. 路由守卫策略

本步完善了 `src/router/guard.js`，当前守卫逻辑如下：

### 未登录访问受保护页面

- 清理权限态
- 跳转到 `/login`
- 带上 `redirect` 参数

### 已登录首次访问受保护页面

- 先补拉当前用户信息
- 再请求 `/admin/base/comm/permmenu`
- 再注入动态路由

### 已登录访问登录页

- 自动回跳到 `redirect` 或 `/dashboard`

### 直接访问有权限但首次未注入的动态路由

- 命中兜底路由
- 守卫检测到该路径属于当前用户可访问路径后重新进入目标路由

### 访问无权限页面

- 跳转 `403`

### 访问不存在页面

- 跳转 `404`

### 5. 菜单渲染

当前侧边栏改为真实权限驱动：

- 不再依赖静态菜单数组
- 使用 `permissionStore.navigationMenus` 渲染
- 根据菜单树结构递归渲染目录与页面项
- 根据后端 `icon` 字段映射图标

涉及文件：

- `src/layout/components/AppSidebar.vue`
- `src/layout/components/AppSidebarItem.vue`
- `src/layout/menu-icons.js`
- `src/layout/menu.js`

补充说明：

- 工作台菜单仍保留为静态保底入口
- 若后端未来返回工作台菜单，可自然与现有逻辑兼容

### 6. 页面级权限控制

页面级权限当前通过“动态路由是否注入 + 可访问路径判断”实现。

这意味着：

- 没有菜单权限的页面，不会被注册为可访问页面
- 用户手动输入 URL 时，也会被路由守卫拦截

这是当前阶段最稳妥、最贴近后端返回结构的做法。

### 7. 按钮级权限控制

本步新增两套按钮权限能力：

### 指令方式

- 指令名：`v-permission`
- 支持值：单个权限码或权限码数组
- 支持参数：
  - 默认 `some`
  - `every` 表示要求全部命中

### 函数方式

- 方法名：`hasPermission`
- 文件：`src/utils/permission.js`

适用场景：

- 模板中直接控制按钮显隐
- JS 中做权限判断

### 8. 未授权与异常页面

本步新增：

- `src/views/error/403.vue`
- 重构 `src/views/error/404.vue`

当前规则：

- 有受保护路径但无权限：进入 `403`
- 路径不存在或业务页面尚未接入：进入 `404`

### 9. 认证态与权限态联动

为了避免动态路由、菜单和按钮权限在登录切换后残留，本步补齐了以下联动：

- 登录失败时清理权限态
- 退出登录时清理权限态
- 请求层收到 `401` 时清理权限态
- 未登录访问受保护路由时清理权限态

涉及文件：

- `src/main.js`
- `src/stores/modules/auth.js`
- `src/http/interceptors.js`
- `src/router/guard.js`

## 当前权限链路

当前完整链路为：

1. 用户登录成功
2. 拉取 `/admin/base/comm/person`
3. 拉取 `/admin/base/comm/permmenu`
4. 生成菜单树、按钮权限与动态路由
5. 渲染侧边栏
6. 用户访问页面时经过页面级权限校验
7. 页面内部按钮再经过按钮级权限校验
